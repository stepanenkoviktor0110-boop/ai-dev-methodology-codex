# Task 6: Create design-retrospective skill

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:skill-master` — [skills/skill-master/SKILL.md](../../../skills/skill-master/SKILL.md)

## Description

Создать процедурный скилл `design-retrospective` для ретроспективы дизайн-сессий. Скилл анализирует фидбек пользователя и коррекции, сделанные во время работы с design-generate/design-system-init, выявляет повторяющиеся эстетические паттерны, записывает уроки в `.design-system/lessons-learned.md`, промоутит уроки с 3+ повторениями в `.design-system/design-principles.md`, и генерирует самодостаточный промт для следующей сессии.

Это аналог скилла `retrospective` (методологическая ретроспектива), но для дизайн-домена. Ключевые отличия: retrospective анализирует процесс разработки (git log, decisions.md, fix rounds), а design-retrospective анализирует эстетические решения (визуальный фидбек, повторяющиеся правки стилей). Выход разный: retrospective пишет в lessons-learned конкретного скилла методологии, а design-retrospective пишет в проектный `.design-system/` и накапливает профиль вкуса проекта.

Decision 11: `design-principles.md` живёт в проекте (`.design-system/`), а не в скилле. Принципы дизайна специфичны для проекта — разные проекты имеют разный эстетический профиль.

## What to do

1. Создать директорию `skills/design-retrospective/`
2. Написать `SKILL.md` — процедурный скилл со следующими фазами:
   - **Phase 1: Collect Evidence** — собрать фидбек пользователя за сессию, список коррекций, изменённые файлы в `.design-system/`
   - **Phase 2: Identify Patterns** — категоризировать повторяющиеся коррекции: цветовые правки, отклонённые шрифтовые пары, корректировки spacing, переработки лейаутов
   - **Phase 3: Write Lessons** — записать уроки в `.design-system/lessons-learned.md` в формате Problem/Cause/Solution/Rule. Если файл не существует — создать. Если существует — дополнить, не дублируя
   - **Phase 4: Promote Principles** — посчитать повторения каждого урока. Если урок появляется 3+ раза — промоутить в `.design-system/design-principles.md` как правило
   - **Phase 5: Generate Next-Session Prompt** — сгенерировать самодостаточный промт для продолжения работы в новой сессии (без зависимости от предыдущего контекста)
3. Каждая фаза должна иметь checkpoint
4. Добавить секцию Self-Verification в конец SKILL.md

## Details

**Files:**
- `skills/design-retrospective/SKILL.md` (new) — процедурный скилл, ~200-280 строк. Создать директорию и файл с нуля.

**Dependencies:**
- Task 1 (shared design-references) — скилл не ссылается на shared references напрямую, но зависимость по волне (Wave 2 после Wave 1). Фактически design-retrospective автономен — он работает с проектными файлами `.design-system/`, а не с reference-файлами скиллов.

**Structure of SKILL.md (mirror retrospective/SKILL.md):**
- Frontmatter: name `design-retrospective`, description с trigger phrases ("дизайн ретроспектива", "design retrospective", "уроки дизайна", "что улучшить в дизайне", "design lessons learned")
- Input: session history (user feedback, corrections), `.design-system/` files
- Output: entries in `.design-system/lessons-learned.md`, promotions to `.design-system/design-principles.md`, next-session prompt
- Language: lessons and communication in Russian (as in retrospective)

**Phase details:**

*Phase 1: Collect Evidence*
- Ask user which design session to analyze (if not obvious from context)
- Read `.design-system/` directory: tokens.json, components/, pages/ — what was created/modified
- Collect user feedback: corrections requested, elements rejected, preferences expressed
- Note files changed during the session

*Phase 2: Identify Patterns*
- Categorize corrections into types: color corrections, rejected font pairs, spacing adjustments, layout reworks, component modifications
- Look for repetitions: same type of correction appearing multiple times = pattern
- If no corrections were made and no patterns detected — report "no patterns detected" and skip to Phase 5

*Phase 3: Write Lessons*
- Check if `.design-system/lessons-learned.md` exists; create if not
- For each identified pattern, write entry in format:
  ```
  ### {YYYY-MM-DD}: {pattern title}
  **Problem:** what went wrong
  **Cause:** why it happened
  **Solution:** how it was fixed
  **Rule:** what to do in the future
  ```
- Check for duplicate entries (same problem already recorded) — skip if exists

*Phase 4: Promote Principles*
- Read all entries in `lessons-learned.md`
- Count occurrences of similar rules (same root cause/same type of correction)
- If a rule appears 3+ times — add to `.design-system/design-principles.md`
- `design-principles.md` format: list of rules with rationale, derived from accumulated lessons
- If file doesn't exist — create with header
- If rule already promoted — skip

*Phase 5: Generate Next-Session Prompt*
- Generate self-contained prompt for continuing design work in a new session
- Prompt includes: project context, current DS state, accumulated principles, what was done, what to focus on next
- Prompt requires no prior conversation context to be useful

**Edge cases:**
- First run (no existing `lessons-learned.md`) — create file from scratch
- Empty session (no corrections, no feedback) — report "no patterns detected", skip lesson creation, still generate next-session prompt
- `.design-system/` directory doesn't exist — refuse with suggestion to run `/design-system-init`
- Duplicate lesson — check before writing, skip if already exists
- `design-principles.md` already contains the promoted rule — skip promotion

**Implementation hints:**
- Mirror the phase/checkpoint structure of `retrospective/SKILL.md` closely, adapting for design domain
- Keep SKILL.md under 500 lines — no references needed for this skill (all logic fits in main file)
- Lesson format matches retrospective's Problem/Cause/Solution/Rule — proven pattern
- The 3+ count for promotion is across all entries in lessons-learned.md, not per session
- Next-session prompt is the key differentiator from methodology retrospective — it enables context-free continuation
