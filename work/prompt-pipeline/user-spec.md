# User Spec: prompt-pipeline

## Что делаем

n8n workflow для пошаговой разработки проектов через copy-paste в бесплатные LLM (ChatGPT, DeepSeek, Qwen). Workflow ведёт пользователя по шагам методологии: от описания идеи до готового набора файлов проекта. На каждом шаге генерируется промт, пользователь копирует его в LLM, вставляет ответ обратно, n8n парсит результат и создаёт файлы. Личный инструмент автора — на случай когда лимиты Claude Code заканчиваются.

## Зачем

Когда лимиты Claude Code заканчиваются, а проект делать надо — нет способа продолжить работу по методологии. Prompt Pipeline даёт альтернативу: тот же пайплайн (спека → декомпозиция → генерация кода), но через любую бесплатную LLM без API-ключей. Цель — от идеи до рабочего проекта за одну сессию.

## Как должно работать

### Основной сценарий

1. Пользователь открывает n8n workflow и запускает его (форма)
2. Форма "Новый проект": выбор шаблона из списка (12 штук) + описание проекта в свободной форме + путь для файлов (настраивается, работает и на VPS и локально)
3. n8n генерирует **промт для спеки** (описание + контекст шаблона) и показывает его
4. Пользователь копирует промт в LLM → получает спеку → вставляет ответ в n8n
5. n8n генерирует **промт для валидации спеки** ("проверь на пробелы и противоречия") и показывает его
6. Пользователь копирует → получает замечания → вставляет → n8n обновляет спеку
7. n8n генерирует **промт для декомпозиции** (разбить на 5-8 файлов MVP, сложные части пометить как заглушки, остальное в бэклог)
8. Пользователь копирует → получает список файлов → вставляет
9. Для каждого файла (шаги 10-12 повторяются):
10. n8n генерирует **промт для файла N** с контекстом: спека + содержимое уже готовых файлов 1..N-1
11. Пользователь копирует → получает код → вставляет
12. (Опционально) Если галочка "Ревью" включена — n8n генерирует промт для code review файла, пользователь копирует → вставляет замечания → n8n показывает промт для исправлений
13. Все файлы готовы → n8n создаёт файлы по указанному пути + предлагает скачать архив
14. Workflow завершён, лог сессии сохранён

### Шаблоны проектов (12 штук)

Шаблоны описаны простым языком — пользователь выбирает по назначению, не по технологии. Стек подбирается автоматически. В форме шаблоны сгруппированы по категориям.

**Сайты**
1. **Визитка / лендинг** — одностраничный сайт с информацией о чём угодно. Пример: "Сайт моей кофейни с меню и контактами"
2. **Сайт с несколькими страницами** — сайт с навигацией и разделами. Пример: "Портфолио фотографа с галереей и страницей 'обо мне'"

**Боты**
3. **Бот-помощник в Telegram** — отвечает на сообщения, выполняет команды. Пример: "Бот, который считает калории по описанию еды"
4. **Бот-уведомлятель в Telegram** — мониторит что-то и присылает алерты. Пример: "Бот, который проверяет курс доллара каждый час и пишет если сильно изменился"

**Веб-приложения и API**
5. **Веб-приложение с интерфейсом** — формы, кнопки, данные. Пример: "Трекер привычек с дашбордом"
6. **API-сервис** — принимает запросы, отдаёт данные, бэкенд без интерфейса. Пример: "Сервис сокращения ссылок с API"

**Автоматизации n8n**
7. **Автоматизация в n8n (webhook)** — реагирует на событие, обрабатывает данные. Пример: "При оплате в Stripe — отправить письмо и записать в таблицу"
8. **Автоматизация в n8n (по расписанию)** — делает что-то регулярно. Пример: "Каждое утро собирает курсы валют и кладёт в Google Sheet"

**Таблицы**
9. **Скрипт для Google Таблиц** — автоматизация внутри таблицы (Apps Script). Пример: "При добавлении строки — отправить уведомление и посчитать итоги"
10. **Макрос для Excel** — автоматизация в Excel (VBA). Пример: "Кнопка, которая собирает данные с нескольких листов в сводный отчёт"

**Утилиты и приложения**
11. **Скрипт-утилита** — запустил, получил результат. Пример: "Скрипт, который переименовывает фото по дате съёмки"
12. **Приложение с окошком** — программа на компьютере, запускается с ярлыка. Пример: "Конвертер файлов — перетащил картинку, получил другой формат"

Каждый шаблон определяет:
- Типичную структуру файлов (5-8 штук)
- Стек (подбирается автоматически)
- Список типичных "сложных мест" для заглушек
- Ограничения размера файлов
- Контекст для промтов

### Заглушки (stubs)

Каждый шаблон знает свои типичные "сложные места". При декомпозиции и генерации кода LLM получает инструкцию: сложные части реализовать как работающие заглушки с пометкой `// TODO:` для доработки позже.

Примеры заглушек по шаблонам:
- **Бот**: `// TODO: подключение к БД — пока данные хранятся в памяти`
- **Веб-приложение**: `// TODO: авторизация — пока захардкожен тестовый пользователь`
- **API-сервис**: `// TODO: валидация входных данных — пока принимаем всё`
- **n8n workflow**: `// TODO: обработка ошибок API — пока логируем и пропускаем`
- **Приложение с окошком**: `# TODO: сохранение настроек — пока значения по умолчанию`
- **Google Sheets**: `// TODO: проверка прав доступа — пока предполагаем что есть`

Заглушки попадают в бэклог как задачи на доработку.

### Формат ответа LLM

В каждом промте есть инструкция для LLM — оборачивать файлы маркерами:

```
=== FILE: path/to/file.ext ===
содержимое файла
=== END FILE ===
```

n8n парсит по маркерам. Если маркеров нет — весь ответ считается одним файлом.

### Восстановление сессии

n8n сохраняет прогресс после каждого шага: номер текущего шага, спека, список файлов, уже сгенерированные файлы. Если пользователь закрыл вкладку или произошёл сбой — при следующем запуске workflow предлагает "Продолжить предыдущий проект" или "Начать новый". При выборе "Начать новый" — предыдущий прогресс удаляется (файлы проекта на диске остаются).

### Логирование

Параллельно с основным flow n8n пишет JSON-лог сессии:
- Выбранный шаблон и описание проекта
- Каждый сгенерированный промт
- Каждый ответ LLM (вставленный пользователем)
- Количество попыток на каждый шаг (повторы)
- Таймстемпы всех шагов
- Итоговая структура файлов
- Заглушки — какие были, какие доработаны

Лог используется для ретроспективы: скармливается в LLM с промтом анализа.

## Критерии приёмки

### Основной flow
- [ ] Workflow запускается в n8n, показывает форму с выбором шаблона, полем описания и полем пути
- [ ] Все 12 шаблонов доступны для выбора, каждый с названием, описанием и примером
- [ ] Путь для файлов задаётся в форме, принимает абсолютный путь (VPS или локальный)
- [ ] Промт для спеки включает описание проекта пользователя и структуру выбранного шаблона
- [ ] Промт для файла N включает текст спеки и полное содержимое файлов 1..N-1
- [ ] Промт для декомпозиции содержит инструкцию: максимум 5-8 файлов, сложные части — заглушки, остальное — бэклог

### Парсинг и файлы
- [ ] Ответ с маркерами `=== FILE: path ===` / `=== END FILE ===` парсится в отдельные файлы
- [ ] Ответ без маркеров сохраняется целиком как один файл
- [ ] Ответ с незакрытым маркером (обрезанный) — сохраняет содержимое до конца ответа, показывает предупреждение
- [ ] Пути файлов из ответа LLM не выходят за пределы директории проекта (path traversal отклоняется)
- [ ] При ошибке записи (нет прав, диск полон) — показывает ошибку с путём, уже записанные файлы не удаляются
- [ ] Файлы создаются по указанному пути с нужной структурой папок (поддиректории создаются автоматически)
- [ ] По завершении доступна кнопка "Скачать архив" — скачивается .zip со всеми файлами проекта

### Управление процессом
- [ ] Пустая вставка (пользователь ничего не вставил) — показывает предупреждение, не переходит к следующему шагу
- [ ] Кнопка "Повторить шаг" показывает тот же промт заново для повторного copy-paste
- [ ] Галочка "Code review" — если включена, после файла показывается дополнительный промт для ревью; если выключена — переход к следующему файлу
- [ ] После шага "спека" показывается промт валидации ("проверь на пробелы и противоречия")
- [ ] Заглушки в коде помечены `// TODO:` с описанием, список заглушек выводится в бэклог по завершении

### Восстановление и логирование
- [ ] Прогресс сохраняется после каждого шага (номер шага, спека, готовые файлы)
- [ ] При запуске workflow, если есть незавершённый проект — форма предлагает "Продолжить" или "Начать новый"
- [ ] JSON-лог сессии записывается автоматически: шаблон, промты, ответы, количество повторов, таймстемпы
- [ ] Промты хранятся в отдельных .md файлах на диске (не внутри n8n нод)

## Ограничения

- Только бесплатные LLM через веб-сайт (chatgpt.com, chat.deepseek.com, Qwen). Без API-ключей.
- Один проект одновременно.
- MVP проекта: максимум 5-8 файлов. Ограничение задаётся шаблоном.
- Размер файлов ограничен шаблоном — все файлы должны помещаться в контекст LLM при подстановке.
- Россия: всё что требует заблокированных API — не используется.

## Риски

- **LLM не следует формату маркеров.** Митигация: кнопка "Повторить шаг" с тем же промтом. В промте — чёткая инструкция с примером формата.
- **Ответ LLM обрезан (контекст переполнен).** Митигация: шаблоны проектируются так, чтобы суммарный контекст (спека + готовые файлы) был компактным. Файлы маленькие по определению шаблона.
- **Качество кода от бесплатных LLM ниже Claude Code.** Митигация: опциональный шаг code review, валидация спеки, заглушки вместо сложной логики. Принимаем как осознанный компромисс.
- **Пользователь вставляет неполный или невалидный ответ.** Митигация: если парсер не находит маркеров — сохраняет как один файл. Кнопка "Повторить шаг" для повторной попытки.
- **Обрыв сессии (закрыл вкладку, сбой VPS).** Митигация: прогресс сохраняется после каждого шага, восстановление при следующем запуске.

## Технические решения

- Платформа — n8n workflow на VPS (уже установлен).
- Промты хранятся в отдельных .md файлах на диске, а не внутри n8n нод — проще редактировать и версионировать.
- Copy-paste вместо API — осознанный выбор: бесплатно, без блокировок, работает с любой LLM.
- Один workflow на весь пайплайн (не набор отдельных workflows).
- Файлы проекта создаются на VPS + опция скачать архив.
- Шаблоны описаны простым языком по назначению, стек подбирается автоматически.
- Заглушки для сложных частей — MVP работает, доработка через бэклог.

## Тестирование

**Unit-тесты:** парсер ответов LLM (маркеры `=== FILE ===`). Тестируемые кейсы: корректные маркеры, отсутствие маркеров, вложенные блоки кода, обрезанный ответ, пути с path traversal.

**Интеграционные тесты:** не делаем — n8n workflow тестируется вручную прогоном.

**E2E тесты:** не делаем — интерфейс это формы n8n, тестировать их автоматически нецелесообразно.

## Как проверить

### Агент проверяет

| Шаг | Инструмент | Ожидаемый результат |
|-----|-----------|-------------------|
| 1. Импортировать workflow в n8n | bash (n8n CLI или API) | Workflow появился в списке |
| 2. Запустить парсер на тестовом ответе с маркерами | bash (node script) | Файлы распарсены, пути корректны |
| 3. Запустить парсер на ответе без маркеров | bash (node script) | Весь ответ сохранён как один файл |
| 4. Запустить парсер на ответе с path traversal | bash (node script) | Путь отклонён, ошибка |
| 5. Проверить что .md файлы промтов на месте | bash (ls) | Все 12 шаблонов + системные промты |

### Пользователь проверяет

- Пройти полный цикл: выбрать "Визитка / лендинг" → описать проект → пройти все шаги copy-paste → убедиться что файлы созданы и лог записан
- Проверить кнопку "Повторить шаг" — вставить мусор вместо ответа, нажать повторить
- Проверить восстановление — закрыть вкладку посередине, запустить workflow заново, выбрать "Продолжить"
- Скачать архив и убедиться что структура файлов корректна
- Проверить что заглушки присутствуют в коде и отражены в бэклоге
