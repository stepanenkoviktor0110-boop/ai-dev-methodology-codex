# Task 12: Pre-deploy QA

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:pre-deploy-qa` — [skills/pre-deploy-qa/SKILL.md](../../../skills/pre-deploy-qa/SKILL.md)

## Description

Финальное приёмочное тестирование фичи design-pipeline. Задача Final Wave -- выполняется после всех Audit Wave задач (code audit, security audit, test audit). Необходимо проверить все acceptance criteria из user-spec и tech-spec, убедиться что все 4 скилла проходят skill-checker, все ссылки на reference-файлы валидны, интеграция design-review в do-task не ломает существующий flow, а like-figma удалён.

Это не код-задача -- здесь не создаётся и не модифицируется код. Задача проверяет результаты работы всех предыдущих задач (1-8) и учитывает замечания аудитов (9-11).

## What to do

1. Прочитать user-spec.md, tech-spec.md, decisions.md -- собрать полный список acceptance criteria и зафиксированных отклонений от плана.

2. Проверить структуру артефактов:
   - 4 директории скиллов: `skills/design-system-init/`, `skills/design-generate/`, `skills/design-review/`, `skills/design-retrospective/`
   - `shared/design-references/` содержит: design-tokens.md, color-psychology.md, color-principles.md, library-catalog.md
   - `skills/design-system-init/references/font-pairing.md` существует
   - `skills/design-generate/references/` содержит: component-patterns.md, generation-guide.md, grid-techniques.md
   - `skills/design-generate/assets/preview-template.html` существует
   - `skills/like-figma/` удалён

3. Запустить skill-checker на каждом из 4 скиллов -- проверить frontmatter, длину (<500 строк), Pattern A/B ссылки, фазы/чекпоинты для procedural скиллов.

4. Валидировать все ссылки на reference-файлы внутри SKILL.md -- каждый путь должен резолвиться в реальный файл.

5. Проверить do-task SKILL.md -- design-review hook присутствует между Step 2.3 и Step 2.4, не нарушает существующий flow (design-review skips silently когда нет DS).

6. Проверить CSP в preview-template.html -- meta tag с `script-src 'none'` присутствует.

7. Проверить валидацию имён файлов компонентов/страниц -- паттерн `/^[a-z0-9-]+$/` описан в скиллах.

8. Сверить каждый acceptance criterion из tech-spec (13 пунктов) и user-spec -- отметить passed/failed/not_verifiable.

9. Записать QA-отчёт в `logs/working/qa-report.json` и краткую запись в decisions.md.

## Verification Steps

QA-задача сама является верификацией. Результат -- qa-report.json со статусом passed/failed.

## Details

**Scope:** Это не кодовая задача. Скиллы -- markdown-инструкции, не исполняемый код. Unit/integration/E2E тесты не применимы. QA проверяет структуру файлов, содержимое, валидность ссылок, соответствие acceptance criteria.

**Файлы для проверки существования:**
- `skills/design-system-init/SKILL.md` -- procedural skill, phases + checkpoints
- `skills/design-generate/SKILL.md` -- procedural skill, phases + checkpoints
- `skills/design-review/SKILL.md` -- informational skill (Decision 4)
- `skills/design-retrospective/SKILL.md` -- procedural skill, phases + checkpoints
- `shared/design-references/design-tokens.md` -- token schema reference
- `shared/design-references/color-psychology.md` -- emotional color map
- `shared/design-references/color-principles.md` -- 10 color principles
- `shared/design-references/library-catalog.md` -- free icon/element libraries catalog
- `skills/design-system-init/references/font-pairing.md` -- font pairing rules (Decision 10: skill-specific, not shared)
- `skills/design-generate/references/component-patterns.md` -- component catalog
- `skills/design-generate/references/generation-guide.md` -- page assembly guide (enriched)
- `skills/design-generate/references/grid-techniques.md` -- 10 grid techniques
- `skills/design-generate/assets/preview-template.html` -- HTML template with CSP

**Файл для проверки отсутствия:**
- `skills/like-figma/` -- должен быть удалён (Task 8)

**do-task integration check:**
- `skills/do-task/SKILL.md` -- найти design-review hook между Step 2.3 (git commit) и Step 2.4 (reviewers)
- Hook должен: проверить `.design-system/tokens.json` + UI files → вызвать design-review
- Guard clauses: не запускать полный pipeline, не генерировать макеты, не проводить интервью, не создавать компоненты

**CSP check (Decision 8):**
- `preview-template.html` должен содержать meta tag: `<meta http-equiv="Content-Security-Policy" content="script-src 'none'; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com">`

**File naming validation (Decision 9):**
- Skills должны описывать паттерн `/^[a-z0-9-]+$/` для имён компонентов и страниц в `.design-system/`

**Reference naming (Decision 7):**
- user-spec упоминает color-theory.md, composition.md, font-pairing.md
- tech-spec маппит: color-theory → color-psychology.md + color-principles.md; composition → grid-techniques.md; font-pairing → font-pairing.md
- При проверке user-spec AC учитывать этот маппинг

**Dependencies:** Tasks 1-8 (создание артефактов), Tasks 9-11 (аудиты -- замечания могли привести к правкам)

**Edge cases:**
- Audit findings (tasks 9-11) могли привести к изменениям -- проверить decisions.md на наличие отклонений
- user-spec AC формулировки отличаются от tech-spec AC -- маппить через Decision 7
- Некоторые user-spec criteria (визуальное качество, реакция на фидбек) -- not_verifiable без live environment
