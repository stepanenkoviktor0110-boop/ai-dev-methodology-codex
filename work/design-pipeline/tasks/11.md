# Task 11: Test Audit

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:test-master` — [skills/test-master/SKILL.md](../../../skills/test-master/SKILL.md)

## Description

Full-feature test quality audit for the design-pipeline feature (L-size, 4 skills).

This is an Audit Wave task -- no code changes, only reading and analysis. The feature uses scenario tests as the primary (and only) testing method, since all skills are markdown instructions with no executable runtime. The scenario tests are designed via the skill-test-designer -> skill-tester pipeline.

The audit reviews scenario test coverage across all 4 design skills (design-system-init, design-generate, design-review, design-retrospective), evaluates whether edge cases, error paths, and boundary conditions are adequately covered, and verifies that the testing strategy is sufficient for an L-size feature. The output is an audit report written to decisions.md.

## What to do

1. Read all 4 SKILL.md files created by tasks 3-6: design-system-init, design-generate, design-review, design-retrospective. Understand the behavior each skill defines.

2. Read the tech-spec Testing Strategy section (scenario tests defined for each skill). This is the baseline of what was planned.

3. Read any existing scenario test files if they were created by earlier tasks (check `work/design-pipeline/` and skill directories for test scenario artifacts).

4. For each of the 4 skills, evaluate scenario test coverage against these dimensions:
   - **Happy path coverage** -- are the core flows tested?
   - **Edge cases** -- are the edge cases from user-spec and tech-spec covered? (e.g., conflicting style sources, corrupted tokens.json, missing components, empty sessions)
   - **Error paths** -- does each skill handle failures gracefully? (e.g., non-web project, no DS exists, invalid JSON)
   - **Boundary conditions** -- context exhaustion, lesson promotion threshold (3+ occurrences), contrast ratio checks
   - **Cross-skill interactions** -- design-review called from do-task, retrospective reading outputs of other skills

5. Assess testing strategy adequacy for L-size feature: is scenario testing alone sufficient? Are there gaps that scenario tests cannot cover?

6. Write audit report to `logs/working/task-11/test-audit-report.md` with findings, gaps identified, and recommendations.

## Details

**Dependencies:** Tasks 3-8 must be completed. All 4 skill SKILL.md files must exist, and the do-task integration (task 7) and like-figma deletion (task 8) must be done.

**Scenario test baseline from tech-spec (22 scenarios total):**

*design-system-init (6 scenarios):*
1. Fresh project with no styles -- creates tokens.json from scratch
2. Project with Tailwind config -- extracts existing values
3. `.design-system/` already exists -- asks update/recreate/cancel
4. Non-web project -- refuses with explanation
5. Conflicting style sources (Tailwind config vs CSS vars) -- shows conflict, asks priority
6. Project with existing CSS vars but no Tailwind -- extracts from vars

*design-generate (7 scenarios):*
1. Valid DS exists -- generates HTML+SVG from description
2. No DS -- refuses, suggests `/design-system-init`
3. Missing simple component (badge) -- creates on-the-fly
4. Missing complex component (carousel) -- decomposes, separate session
5. Context exhaustion -- stops, generates continuation prompt
6. Corrupted/invalid tokens.json -- error with suggestion to recreate
7. SVG renders incorrectly -- asks for screenshot, simplifies problematic elements

*design-review (4 scenarios):*
1. UI files with hardcoded colors -- gives concrete token recommendations
2. Files already using DS tokens -- reports "styles match DS"
3. No `.design-system/tokens.json` -- skips silently
4. No UI files changed -- skips silently

*design-retrospective (5 scenarios):*
1. Session with corrections -- creates lessons in lessons-learned.md
2. Lesson repeated 3+ times -- promotes to design-principles.md
3. Generates next-session prompt
4. First run (no existing lessons-learned.md) -- creates file from scratch
5. Empty session (no corrections) -- reports "no patterns detected", skips lesson creation

**Audit dimensions to evaluate:**

- Coverage completeness: do the 22 scenarios cover all behaviors defined in each SKILL.md?
- Missing edge cases: what behaviors in the skills are NOT covered by any scenario?
- Error path coverage: does each skill have at least one error/failure scenario?
- Boundary conditions: are threshold values tested (contrast ratio 4.5:1, lesson count 3+, context limits)?
- Cross-skill dependencies: is the do-task -> design-review integration path tested? Is retrospective's consumption of other skill outputs tested?
- Negative testing: are invalid inputs, malformed data, and unauthorized operations covered?
- Testing strategy fit: for an L-size feature with 4 skills and no executable code, is scenario testing alone sufficient? What risks remain uncovered?

**Output format:** Audit report in `logs/working/task-11/test-audit-report.md` following the standard format. Include: summary of findings, per-skill coverage assessment, identified gaps, risk assessment, and recommendations for additional scenarios if needed.
