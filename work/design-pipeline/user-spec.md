# User Spec: design-pipeline

## Что делаем

Создаём Design Pipeline — фреймворк из 9 скиллов для работы над визуальной эстетикой веб-проектов. Агент работает как стилист: по запросу включается в работу для наведения красоты — подбирает палитры, сочетания цветов, шрифты, выстраивает визуальную согласованность. Фреймворк аналогичен dev-методологии по структуре: от создания дизайн-системы через интервью до генерации макетов, ревью, ретроспективы и управления сессиями. Всё без Figma API — дизайн-система хранится как код в проекте.

## Зачем

Агент пишет код, но не концентрируется на эстетике. Нет инструмента для целенаправленной работы над красотой: подбор палитр под контекст проекта, сочетания цветов, стилевая согласованность между экранами, подбор шрифтов. Разработчик без дизайнера и без Figma получает возможность делать визуально качественные интерфейсы через AI-агента, который учится на фидбеке и накапливает "вкус" проекта.

## Как должно работать

### Сценарий А: Создание дизайн-системы

1. Пользователь вызывает `/design-system-init`
2. Агент сканирует проект: находит цвета, шрифты, spacing в коде (Tailwind, CSS, переменные)
3. Проводит интервью: предлагает конкретные значения, пользователь подтверждает/корректирует. Темы: палитра (primary, secondary, neutrals, semantic), шрифты (Google Fonts, пары headings+body), spacing, radii, shadows, компоненты
4. Создаёт `.design-system/` в проекте: `tokens.json` + HTML-файлы компонентов
5. Каждый компонент можно открыть в браузере и посмотреть
6. Пользователь проверяет результат в браузере

### Сценарий Б: Генерация макета

1. Пользователь описывает: "сделай макет страницы логина"
2. Агент читает дизайн-систему, уточняет если неоднозначно
3. Собирает страницу из существующих компонентов
4. Генерит HTML (интерактивный, в браузере) + SVG (статика)
5. Пользователь открывает HTML, присылает скрин с комментариями
6. Агент итерирует. Большие страницы — декомпозирует на секции, делает в разных сессиях

### Сценарий В: Наведение красоты (MVP-интеграция с методологией)

Точечные обращения к design-pipeline из dev-методологии. Ключевой принцип: **не перегружать и без того прожорливую методологию**. Обращения минимальные по токенам.

**Когда срабатывает:** do-task видит UI-задачу + `.design-system/` существует в проекте.

**Что происходит:**
1. design-review читает ТОЛЬКО `tokens.json` (не все компоненты, не references — экономия контекста)
2. Сканирует изменённые файлы задачи (не весь проект)
3. Даёт 2-3 точечные рекомендации: "цвет кнопки не из палитры → --color-primary-500", "gap 8px → DS минимум 16px"
4. Применяет. Всё.

**Чего НЕ происходит:**
- Не запускается полный пайплайн
- Не генерируются макеты
- Не проводится интервью
- Не запускается ретроспектива
- Не создаются новые компоненты

**Оценка стоимости:** ~500-1000 токенов сверх обычного do-task (чтение tokens.json + 2-3 рекомендации). Это ~1-2% от стоимости задачи.

### Сценарий Г: Ретроспектива

1. После нескольких генераций/ревью — агент собирает фидбек
2. Анализирует: какие правки повторялись, что не нравилось
3. Создаёт записи в `lessons-learned.md`
4. Если урок повторяется 3+ раза — промоутит в `design-principles.md` как правило
5. Генерирует промт для следующей сессии (не запускает автоматически)

## Критерии приёмки

- [ ] `/design-system-init` создаёт `.design-system/tokens.json` с цветами, шрифтами (Google Fonts), spacing
- [ ] `/design-system-init` создаёт минимум 3 компонента (button, card, input) как HTML-файлы
- [ ] Каждый компонент открывается в браузере автономно и выглядит корректно
- [ ] `/design-generate` по описанию создаёт `.design-system/pages/{name}.html` и `.svg`
- [ ] HTML использует токены из DS (нет hardcoded цветов/размеров)
- [ ] Contrast ratio текста ≥ 4.5:1
- [ ] При отсутствии дизайн-системы — отказ с предложением `/design-system-init`
- [ ] При не-веб проекте — отказ с объяснением
- [ ] Design-review в do-task даёт конкретные рекомендации (не "сделай красивее", а "используй --color-primary-500")
- [ ] Ретроспектива после 3+ итераций фидбека создаёт записи в `lessons-learned.md`
- [ ] Повторяющийся урок (3+ раза) промоутится в `design-principles.md`
- [ ] При закрытии сессии генерируется промт для следующей

## Ограничения

- **Без Figma API** — всё хранится как код в `.design-system/` проекта
- **MVP: только веб-проекты** — не-веб получают отказ с объяснением. Поддержка других типов — post-MVP
- **Шрифты обязательны** — Google Fonts, подбираются под контекст проекта
- **Каталог библиотек** — ссылки на бесплатные библиотеки иконок/элементов с описанием специфики. Не хранилище стоков — только ссылки и описания
- **Нет дизайн-системы = нет работы** — строгий отказ, предложение создать через `/design-system-init`
- **Экономия токенов** — максимальная. MVP-интеграция только средний уровень (design-review). Лёгкий (токены при write-code) и полный (design-generate при do-task) — post-MVP
- **Большие задачи** — декомпозируются на композиции и выполняются в разных сессиях (как в методологии)

## Риски

- **Качество генерации без браузерного рендеринга.** Агент генерит HTML/SVG без возможности увидеть результат. **Митигация:** пользователь присылает промежуточные скрины с комментариями, агент итерирует.
- **Сжигание токенов.** Полный пайплайн на большом проекте может исчерпать контекст. **Митигация:** MVP только средний уровень интеграции, декомпозиция больших задач на сессии.
- **Вкусовщина.** Агент может считать красивым то, что пользователь считает уродливым. **Митигация:** пользователь присылает референсы, ретроспектива накапливает профиль вкуса, промоушен уроков в принципы.

## Технические решения

- Мы решили создать **9 самостоятельных скиллов** в `skills/design-pipeline/`, потому что монолитный скилл (like-figma) оказался слишком большим и не поддерживает сессии/ретроспективы.
- Мы решили сделать **design-review самостоятельным скиллом** (не субагентом), потому что он вызывается из разных контекстов: design-execute, do-task (MVP-интеграция), напрямую.
- Мы решили **удалить like-figma** после миграции, потому что пайплайн полностью его заменяет. References и assets переиспользуются.
- Мы решили **не поддерживать не-веб проекты в MVP**, потому что это отдельная задача с другими паттернами.
- Мы решили **не делать лёгкий и полный уровни интеграции в MVP**, потому что нужно сначала проверить средний и не тратить токены.
- Мы решили что агент **спрашивает пользователя при нарушении логики** (например, тёмная тема без dark-токенов), а не принимает решение сам.
- Мы решили что **сложные компоненты** (carousel, data-table) декомпозируются на части, а **простые** (badge, alert) создаются на лету.

## Тестирование

**Unit-тесты:** не применимы — скиллы являются markdown-инструкциями, не кодом.

**Сценарные тесты:** через skill-test-designer → skill-tester. Основной метод валидации.

**Ручная проверка:** на реальном веб-проекте (кальянка или аналогичный). Пользователь оценивает визуальный результат в браузере.

## Как проверить

### Агент проверяет

| Шаг | Инструмент | Ожидаемый результат |
|-----|-----------|-------------------|
| 1. Проверить что tokens.json валидный JSON | bash (node -e / python -c) | Парсится без ошибок, содержит colors, typography, spacing |
| 2. Проверить что HTML-компоненты содержат CSS custom properties | grep | Нет hardcoded hex/px, используются --color-*, --space-*, --font-* |
| 3. Проверить contrast ratio | bash (скрипт проверки) | Все текстовые пары ≥ 4.5:1 |
| 4. Проверить что design-review даёт конкретные рекомендации | skill-tester | Рекомендации содержат конкретные токены, не общие фразы |

### Пользователь проверяет

- Открыть HTML-компоненты в браузере — визуально выглядят корректно и эстетично
- Открыть сгенерированную страницу — лейаут соответствует описанию, цвета/шрифты согласованы
- Прислать скрин с комментариями — агент адекватно реагирует на фидбек
- Проверить что ретроспектива создала осмысленные уроки, а не формальные отписки

## Post-MVP

- Лёгкий уровень интеграции: при write-code агент читает токены из DS и применяет
- Полный уровень: design-generate при do-task для новых экранов
- Поддержка не-веб проектов: CLI-стилизация, бот-интерфейсы
- Автоматический аудит всего проекта на соответствие DS
