# User Spec: design-pipeline

## Что делаем

Создаём Design Pipeline — фреймворк из 9 скиллов для работы над визуальной эстетикой веб-проектов. Агент работает как стилист: по запросу включается в работу для наведения красоты — подбирает палитры, сочетания цветов, шрифты, выстраивает визуальную согласованность. Фреймворк аналогичен dev-методологии по структуре: от создания дизайн-системы через интервью до генерации макетов, ревью, ретроспективы и управления сессиями. Всё без Figma API — дизайн-система хранится как код в проекте.

## Зачем

Агент пишет код, но не концентрируется на эстетике. Нет инструмента для целенаправленной работы над красотой: подбор палитр под контекст проекта, сочетания цветов, стилевая согласованность между экранами, подбор шрифтов. Разработчик без дизайнера и без Figma получает возможность делать визуально качественные интерфейсы через AI-агента, который учится на фидбеке и накапливает "вкус" проекта.

## Как должно работать

### Сценарий А: Создание дизайн-системы

1. Пользователь вызывает `/design-system-init`
2. Агент сканирует проект: находит цвета, шрифты, spacing в коде (Tailwind, CSS, переменные)
3. Проводит интервью: сначала выясняет контекст проекта (для кого, какое настроение, формальный/неформальный), затем на основе эмоциональной карты цветов предлагает подходящую палитру. Темы: контекст и настроение → палитра (primary, secondary, neutrals, semantic), шрифтовые пары (Google Fonts, headings+body), spacing (с учётом золотого сечения/Фибоначчи), radii, shadows, композиционная сетка, компоненты
4. Создаёт `.design-system/` в проекте: `tokens.json` + HTML-файлы компонентов
5. Каждый компонент можно открыть в браузере и посмотреть
6. Пользователь проверяет результат в браузере

**Edge cases сценария А:**
- Конфликтующие значения в проекте (Tailwind конфиг говорит одно, CSS переменные — другое) → агент показывает конфликт, спрашивает что приоритетнее
- Проект без CSS/стилей вообще → агент предлагает дефолтные значения, интервью идёт с нуля
- `.design-system/` уже существует → спросить: обновить/пересоздать/отменить

### Сценарий Б: Генерация макета

1. Пользователь описывает: "сделай макет страницы логина"
2. Агент читает дизайн-систему, уточняет если неоднозначно
3. Собирает страницу из существующих компонентов
4. Генерит HTML (интерактивный, открывается в браузере) + SVG (статика — для вставки в документацию, презентации, обсуждения без браузера)
5. Пользователь открывает HTML, присылает скрин с комментариями
6. Агент итерирует. Если SVG/HTML рендерится криво — просит скрин, корректирует
7. Большие страницы — декомпозирует на секции, делает в разных сессиях
8. Если токены контекста заканчиваются — останавливается, генерирует промт для продолжения в новой сессии

**Edge cases сценария Б:**
- tokens.json повреждён или невалидный JSON → ошибка с предложением пересоздать через `/design-system-init`
- Запрошенный компонент отсутствует в DS: простой (badge) → создать на лету; сложный (carousel) → декомпозиция + отдельная сессия
- SVG рендерит некорректно (тени, градиенты, шрифты) → попросить скрин, упростить проблемные элементы

### Сценарий В: Наведение красоты (MVP-интеграция с методологией)

Точечные обращения к design-pipeline из dev-методологии. Ключевой принцип: **не перегружать и без того прожорливую методологию**. Обращения минимальные по токенам.

**Когда срабатывает:** do-task видит UI-задачу + `.design-system/tokens.json` существует в проекте.

**Как вызывается:** do-task SKILL.md получает дополнение — после завершения кодирования (перед ревью) проверить наличие `.design-system/tokens.json`. Если есть и задача содержит UI-изменения (файлы .tsx/.vue/.html/.css) → вызвать design-review как субагента. Это одна строка в do-task SKILL.md, не переписывание скилла.

**Что происходит:**
1. design-review читает ТОЛЬКО `tokens.json` (не все компоненты, не references — экономия контекста)
2. Сканирует изменённые файлы задачи (не весь проект)
3. Даёт 2-3 точечные рекомендации: "цвет кнопки не из палитры → --color-primary-500", "gap 8px → DS минимум 16px"
4. Применяет. Всё.

**Edge cases сценария В:**
- DS существует, но задача не содержит UI-файлов → не вызывается
- design-review нашёл 0 замечаний → просто пишет "стили соответствуют DS" и не тратит токены на итерации

**Чего НЕ происходит:**
- Не запускается полный пайплайн
- Не генерируются макеты
- Не проводится интервью
- Не запускается ретроспектива
- Не создаются новые компоненты

**Оценка стоимости:** ~500-1000 токенов сверх обычного do-task (чтение tokens.json + 2-3 рекомендации). Это ~1-2% от стоимости задачи.

### Сценарий Г: Ретроспектива

1. После нескольких генераций/ревью — агент собирает фидбек
2. Анализирует: какие правки повторялись, что не нравилось
3. Создаёт записи в `lessons-learned.md`
4. Если урок повторяется 3+ раза — промоутит в `design-principles.md` как правило
5. Генерирует промт для следующей сессии (не запускает автоматически)

## Критерии приёмки

- [ ] `/design-system-init` создаёт `.design-system/tokens.json` с цветами, шрифтами (Google Fonts), spacing
- [ ] `/design-system-init` создаёт минимум 3 компонента (button, card, input) как HTML-файлы
- [ ] Каждый компонент открывается в браузере автономно (HTML валиден, CSS custom properties резолвятся)
- [ ] `/design-generate` по описанию создаёт `.design-system/pages/{name}.html` и `.svg`
- [ ] HTML использует токены из DS (нет hardcoded цветов/размеров)
- [ ] Contrast ratio текста ≥ 4.5:1
- [ ] При кривом рендеринге SVG/HTML — агент запрашивает скрин и итерирует
- [ ] При исчерпании контекста — агент останавливается и генерирует промт для продолжения
- [ ] При отсутствии дизайн-системы — отказ с предложением `/design-system-init`
- [ ] При не-веб проекте — отказ с объяснением
- [ ] design-system-init учитывает контекст проекта при подборе палитры (спокойный сайт → не агрессивные цвета)
- [ ] Shared references содержат color-theory.md, composition.md, font-pairing.md с конкретными правилами
- [ ] Design-review в do-task даёт конкретные рекомендации (не "сделай красивее", а "используй --color-primary-500")
- [ ] Ретроспектива после завершения дизайн-сессии (1+ генерация + фидбек от пользователя) создаёт записи в `lessons-learned.md`
- [ ] Повторяющийся урок (3+ раза) промоутится в `design-principles.md`
- [ ] При закрытии сессии генерируется промт для следующей

## Ограничения

- **Без Figma API** — всё хранится как код в `.design-system/` проекта
- **MVP: только веб-проекты** — не-веб получают отказ с объяснением. Поддержка других типов — post-MVP
- **Шрифты обязательны** — Google Fonts, подбираются под контекст проекта
- **Каталог библиотек** — ссылки на бесплатные библиотеки иконок/элементов с описанием специфики. Не хранилище стоков — только ссылки и описания
- **Нет дизайн-системы = нет работы** — строгий отказ, предложение создать через `/design-system-init`
- **Экономия токенов** — максимальная. MVP-интеграция только средний уровень (design-review). Лёгкий (токены при write-code) и полный (design-generate при do-task) — post-MVP
- **Большие задачи** — декомпозируются на композиции и выполняются в разных сессиях (как в методологии)

## Риски

- **Качество генерации без браузерного рендеринга.** Агент генерит HTML/SVG без возможности увидеть результат. **Митигация:** пользователь присылает промежуточные скрины с комментариями, агент итерирует.
- **Сжигание токенов.** Полный пайплайн на большом проекте может исчерпать контекст. **Митигация:** MVP только средний уровень интеграции, декомпозиция больших задач на сессии.
- **Вкусовщина.** Агент может считать красивым то, что пользователь считает уродливым. **Митигация:** пользователь присылает референсы, ретроспектива накапливает профиль вкуса, промоушен уроков в принципы.

## Технические решения

- Мы решили **MVP из 4 скиллов** (design-system-init, design-generate, design-review, design-retrospective), потому что ретроспектива — источник развития проекта, без неё качество не растёт. Полный пайплайн из 9 скиллов — post-MVP v2.
- Мы решили сделать **design-review самостоятельным скиллом** (не субагентом), потому что он вызывается из разных контекстов: design-execute, do-task (MVP-интеграция), напрямую.
- Мы решили **удалить like-figma** после миграции, потому что пайплайн полностью его заменяет. References и assets переиспользуются.
- Мы решили **design-retrospective — отдельный скилл**, а не переиспользование `/retrospective`, потому что: (а) `/retrospective` анализирует процесс разработки (git log, decisions.md, fix rounds), а design-retrospective анализирует эстетические решения (фидбек на визуал, повторяющиеся правки стилей); (б) выход разный — `/retrospective` пишет в lessons-learned конкретного скилла методологии, а design-retrospective пишет в `.design-system/lessons-learned.md` и промоутит в `design-principles.md`; (в) creative focus — дизайн-ретроспектива собирает профиль вкуса проекта, а не процессные улучшения.
- Мы решили **SVG генерировать параллельно с HTML**, потому что SVG нужен для вставки в документацию, презентации, обсуждения в чатах — контексты где браузер недоступен. HTML — для интерактивной проверки.
- Мы решили **не поддерживать не-веб проекты в MVP**, потому что это отдельная задача с другими паттернами.
- Мы решили **связку с методологией делать точечной** (~500-1000 токенов): design-review читает только tokens.json и сканирует только изменённые файлы задачи. Лёгкий и полный уровни — post-MVP.
- Мы решили что агент **спрашивает пользователя при нарушении логики** (например, тёмная тема без dark-токенов), а не принимает решение сам.
- Мы решили что **сложные компоненты** (carousel, data-table) декомпозируются на части, а **простые** (badge, alert) создаются на лету.

## Тестирование

**Unit-тесты:** не применимы — скиллы являются markdown-инструкциями, не кодом.

**Интеграционные тесты:** не делаем — нет кода для интеграционного тестирования. Скиллы тестируются сценарно.

**E2E тесты:** не делаем — скиллы не имеют runtime, который можно протестировать end-to-end.

**Сценарные тесты (основной метод):** через skill-test-designer → skill-tester. Проверяют что скилл корректно реагирует на типовые запросы.

**Ручная проверка:** на реальном веб-проекте (кальянка или аналогичный). Пользователь оценивает визуальный результат в браузере.

## Как проверить

### Агент проверяет

| Шаг | Инструмент | Ожидаемый результат |
|-----|-----------|-------------------|
| 1. Проверить что tokens.json валидный JSON | bash (node -e / python -c) | Парсится без ошибок, содержит colors, typography, spacing |
| 2. Проверить что HTML-компоненты содержат CSS custom properties | grep | Нет hardcoded hex/px, используются --color-*, --space-*, --font-* |
| 3. Проверить contrast ratio | bash (скрипт проверки) | Все текстовые пары ≥ 4.5:1 |
| 4. Проверить что design-review даёт конкретные рекомендации | skill-tester | Рекомендации содержат конкретные токены, не общие фразы |

### Пользователь проверяет

- Открыть HTML-компоненты в браузере — визуально выглядят корректно и эстетично
- Открыть сгенерированную страницу — лейаут соответствует описанию, цвета/шрифты согласованы
- Прислать скрин с комментариями — агент адекватно реагирует на фидбек
- Проверить что ретроспектива создала осмысленные уроки, а не формальные отписки

## Roadmap

### MVP (4 скилла — базовый цикл + обучение)

| Скилл | Что делает |
|-------|-----------|
| design-system-init | Интервью → tokens.json + компоненты |
| design-generate | Описание → HTML + SVG макет |
| design-review | Точечные рекомендации при UI-задачах в do-task (~500-1000 токенов) |
| design-retrospective | Анализ сессии → уроки → промоушен в принципы → промт для следующей сессии |

Shared references (смысловое ядро — дизайнерские знания, вшитые в скиллы):

| Reference | Содержание |
|-----------|-----------|
| design-tokens.md | Схема токенов, naming conventions, CSS custom properties |
| component-patterns.md | Шаблоны компонентов, BEM, variant system |
| generation-guide.md | Лейауты, паттерны сборки страниц |
| design-principles.md | Базовые принципы + накопленные уроки (пополняется ретроспективой) |
| library-catalog.md | Каталог бесплатных библиотек иконок/элементов с описанием специфики |
| **color-theory.md** | **Теория цвета:** цветовые гармонии (комплементарные, аналогичные, триада, split-complementary), правила сочетаний, contrast ratio, цветовые пропорции (60-30-10). **Эмоциональное считывание цветов:** карта "контекст проекта → подходящая палитра" (спокойный семейный сайт → мягкие тёплые тона, НЕ кричащие/агрессивные; корпоративный → сдержанные; креативное агентство → смелые). Для каждого основного цвета — базовое понимание контекста применимости (когда пользователь не знает что выбрать) |
| **composition.md** | **Пропорции и сетки:** золотое сечение (1:1.618), последовательность Фибоначчи в лейаутах и spacing, rule of thirds. **Интересные сетки:** asymmetric grids, modular grids, baseline grids. **Композиция:** визуальная иерархия, focal points, whitespace как инструмент, ритм и повторение |
| **font-pairing.md** | Правила подбора шрифтовых пар, классические комбинации, весовые контрасты, типографическая шкала |

Связка с методологией: design-review вызывается из do-task при UI-задачах. design-retrospective вызывается при завершении сессии.

### Post-MVP v1 (углубление обучения)

Когда базовый цикл и ретроспектива стабильны — углубляем:

- **Профиль вкуса проекта** — накапливается из референсов пользователя + lessons-learned
- **Автоматический промоушен** — урок повторяется 3+ раз → правило в design-principles.md
- **Расширенное управление сессиями** — анализ текущей → боли → лучшие практики → промт
- Лёгкий уровень интеграции: при write-code читает токены из DS

### Post-MVP v2 (полный пайплайн)

Когда качество стабильно — масштабируем оркестрацию:

- **design-spec** — что рисуем, для кого, какие экраны
- **design-plan** — лейауты, решения, breakpoints
- **design-decompose** — разбивка на атомарные задачи
- **design-execute** — оркестрация сессий, волны
- **design-done** — финализация, архив
- Полный уровень интеграции: design-generate при do-task для новых экранов
- Поддержка не-веб проектов
- Автоматический аудит проекта на соответствие DS
