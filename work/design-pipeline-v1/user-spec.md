# User Spec: design-pipeline-v1

## Что делаем

Углубляем обучение Design Pipeline — добавляем накопление дизайнерского опыта между сессиями и проектами. Агент запоминает предпочтения проекта (taste-profile), переносит опыт между проектами через категоризацию, подбирает стилевые рецепты по контексту, и объясняет дизайн-решения понятным языком. Дополнительно: лёгкая интеграция с code-writing для применения токенов при написании UI-кода.

## Зачем

Разработчик без дизайнера хочет получать от агента не "аккуратные" предложения, а смелые решения с характером. Сейчас MVP Design Pipeline даёт безопасные/банальные варианты — нет "люкса". Описания трендов и стилей написаны на языке агента — приходится уточнять. Агент не помнит предпочтения между сессиями: каждый раз начинает с чистого листа. Опыт из одного проекта не помогает в другом. В результате агент не растёт как дизайнер.

## Как должно работать

### Сценарий А: Накопление вкуса проекта

1. Пользователь проводит сессию design-generate, итерирует макет
2. По завершении запускает `/design-retrospective`
3. Ретроспектива анализирует сессию: что пользователь принял, что отклонил, какие корректировки делал
4. Записывает в `.design-system/taste-profile.md`: предпочтения по цветам, типографике, стилю, уровню смелости, антипаттерны ("отклонил градиенты 2 раза")
5. При следующем запуске design-generate агент читает taste-profile и предлагает варианты, учитывая накопленные предпочтения
6. Через 3 сессии агент предлагает из предпочтений без напоминаний

**Edge cases:**
- taste-profile.md не существует (первая сессия) → создать при первой ретроспективе
- Конфликтующие предпочтения между сессиями ("сессия 1: тёплые тона", "сессия 3: холодные") → latest wins, старое предпочтение помечается как пересмотренное
- taste-profile.md повреждён → пересоздать при следующей ретроспективе с предупреждением

### Сценарий Б: Кросс-проектный опыт

1. Пользователь инициализирует DS в новом проекте через `/design-system-init`
2. Агент определяет категорию проекта на интервью (landing, webapp, admin, portfolio, другое)
3. Читает секцию этой категории из `shared/design-references/designer-experience.md`
4. Использует накопленный опыт для предложений: "В прошлых webapp-проектах хорошо работали приглушённые нейтральные палитры с одним ярким акцентом"
5. При ретроспективе — обобщённые выводы сессии записываются обратно в секцию категории

**Edge cases:**
- Категория проекта новая (нет секции в experience) → работать без опыта, создать секцию при первой ретроспективе
- designer-experience.md не существует → создать при первой ретроспективе
- Опыт из одной категории не релевантен → агент читает только свою секцию, не всё

### Сценарий В: Контекстные стилевые рецепты

1. При `/design-system-init` агент выясняет настроение проекта на интервью (как в MVP)
2. По результату интервью подбирает подходящий стилевой профиль из `shared/design-references/style-profiles.md` (luxury, brutalist, editorial, minimal, playful и т.д.)
3. Профиль содержит конкретные дизайнерские приёмы: "luxury = большие пустоты, serif заголовки, приглушённые цвета, тонкие линии, асимметрия"
4. Агент использует рецепт как отправную точку для предложений, адаптируя под конкретный проект

**Edge cases:**
- Проект не вписывается ни в один профиль → комбинировать элементы из нескольких, спросить пользователя
- Пользователь хочет стиль, которого нет в профилях → использовать общие принципы, записать новый стиль в experience

### Сценарий Г: Понятные описания

1. Агент описывает дизайн-решение в любом дизайн-скилле (init, generate, review, retrospective)
2. Сначала объясняет на языке пользователя: "тёплые приглушённые тона с одним ярким акцентом"
3. Затем техническое обоснование: "split-complementary palette, десатурированная база HSL(35, 15%, 85%) с акцентом HSL(210, 70%, 45%)"
4. Пользователь понимает предложение без уточнений

**Edge cases:**
- Техническое обоснование невозможно (чисто эстетическое решение) → только человеческий слой, без принудительного техобоснования
- design-review (lightweight, cap 3 рекомендации) → двухслойность не обязательна, достаточно формата "found X → use Y" — добавлять объяснение только при неочевидной замене

### Сценарий Д: Code-writing интеграция

1. Разработчик пишет UI-код через code-writing
2. Если `.design-system/tokens.json` существует И задача содержит UI-файлы → агент читает tokens.json
3. При написании CSS/стилей использует CSS custom properties из DS вместо хардкода
4. Если условия не выполнены → silent skip, никакого overhead

**Edge cases:**
- tokens.json невалидный → silent skip, не блокировать кодирование
- Файл не UI (.py, .sql) → не активируется

## Критерии приёмки

- [ ] design-retrospective создаёт/обновляет `.design-system/taste-profile.md` с секциями: цветовые предпочтения, типографика, стиль, уровень смелости (conservative/balanced/bold/experimental), антипаттерны
- [ ] design-generate SKILL.md Phase 0 содержит шаг: читать `.design-system/taste-profile.md` (if exists), передать предпочтения как контекст в Phase 2 (layout selection) и Phase 3 (assembly)
- [ ] design-system-init читает taste-profile при повторной инициализации (update) и предлагает значения на основе накопленных предпочтений
- [ ] Создан `shared/design-references/designer-experience.md` с категориями проектов (landing, webapp, admin, portfolio) и структурой секций (предпочтения, что работало, антипаттерны)
- [ ] design-system-init читает секцию по категории проекта при интервью и использует опыт для предложений
- [ ] design-retrospective записывает обобщённые выводы в секцию категории designer-experience.md
- [ ] Создан `shared/design-references/style-profiles.md` с минимум 6 стилевыми профилями (luxury, brutalist, editorial, minimal, playful, corporate) — каждый с конкретными приёмами
- [ ] design-system-init подбирает стилевой профиль по результатам интервью (настроение + категория) — SKILL.md содержит шаг сопоставления результатов интервью с профилями и инструкцию использовать приёмы из выбранного профиля при предложении палитры/типографики/spacing
- [ ] Все 4 дизайн-скилла содержат инструкцию двухслойного описания (design-review — опционально, только при неочевидных заменах)
- [ ] code-writing SKILL.md содержит условный шаг в Phase 1.2: если tokens.json существует И файлы UI → читать токены, использовать CSS custom properties
- [ ] При отсутствии tokens.json или не-UI файлах → silent skip, нулевой overhead
- [ ] Все ссылки на новые reference-файлы в SKILL.md резолвятся в существующие файлы
- [ ] При повреждённом taste-profile.md → design-retrospective пересоздаёт файл с предупреждением пользователю
- [ ] При отсутствии или невалидном designer-experience.md → design-system-init работает без опыта, не падает
- [ ] designer-experience.md: запись append-only (добавление в конец секции категории), не перезапись всего файла — безопасно для последовательных проектов
- [ ] Нет регрессии: все существующие фазы, чекпоинты и reference-ссылки в 4 дизайн-скиллах сохранены; новые шаги добавлены, а не заменяют старые

## Ограничения

- **Token budget:** overhead на задачу ~1000-1500 токенов. Читать только релевантные секции (одна категория experience, один style-profile).
- **Markdown-only:** все артефакты — markdown-файлы. Нет JSON-конфигов для taste-profile или experience.
- **Только веб-проекты:** как и в MVP, не-веб проекты получают отказ.
- **Обратная совместимость:** существующие `.design-system/` проектов не затрагиваются до явного запуска ретроспективы. Новые файлы создаются при первом использовании.

## Риски

- **Контаминация опыта:** опыт из одного проекта искажает рекомендации для другого. **Митигация:** категоризация по типам проектов, агент читает только свою секцию.
- **Раздувание контекста:** style-profiles.md становится слишком большим. **Митигация:** контекстный подбор — агент читает только подходящий профиль, не весь файл.
- **Конфликты предпочтений:** taste-profile содержит противоречивые записи. **Митигация:** latest wins — новая сессия перезаписывает старые предпочтения, пересмотренное помечается.

## Технические решения

- Мы решили хранить taste-profile в `.design-system/taste-profile.md` (в проекте), потому что это проектно-специфичные данные.
- Мы решили хранить designer-experience в `shared/design-references/designer-experience.md` (глобально), потому что это кросс-проектный опыт, доступный при init любого проекта.
- Мы решили категоризировать experience по типам проектов (landing, webapp, admin, portfolio), потому что это позволяет читать только релевантную секцию и избежать контаминации.
- Мы решили использовать markdown (не JSON) для taste-profile, потому что агент лучше работает с текстом в свободной форме для субъективных предпочтений.
- Мы решили добавлять двухслойные описания во все 4 дизайн-скилла, потому что пользователь жаловался на непонятность технических описаний.
- Мы решили интегрировать code-writing через условный шаг в Phase 1.2 (не субагент), потому что overhead субагента слишком велик для лёгкого уровня интеграции.
- Мы решили, что style-profiles подбираются по контексту интервью (не грузятся все), потому что это экономит токены и предотвращает нерелевантные предложения.
- Мы решили, что designer-experience.md работает в режиме append-only (добавление в секцию категории), потому что файл глобальный и перезапись может потерять данные от предыдущих проектов.
- Мы решили, что двухслойные описания в design-review опциональны (только при неочевидных заменах), потому что скилл ограничен 3 рекомендациями в формате "found X → use Y" и лишний текст раздувает overhead.

## Тестирование

**Unit-тесты:** не применимы — markdown-only feature без исполняемого кода.

**Интеграционные тесты:** не делаем — нечего интегрировать программно.

**E2E тесты:** не делаем — нет UI или API для end-to-end проверки.

**Scenario testing:** основной метод (как в MVP). Сценарии проверяют: создание/обновление taste-profile, категоризация experience, подбор style-profile, двухслойные описания, code-writing интеграция, silent skip при отсутствии DS, обработка повреждённых файлов.

### Пользователь проверяет
- Запустить `/design-system-init` в тестовом проекте — агент должен спросить категорию и предложить стилевой профиль
- После 2-3 сессий `/design-generate` + `/design-retrospective` — проверить что taste-profile.md появился и содержит реальные предпочтения
- Описания решений понятны без уточнений
